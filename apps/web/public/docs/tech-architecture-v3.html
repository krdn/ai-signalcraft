🏗️ 수집 인터페이스 (Collector)
interface Collector<T = unknown> {
  readonly source: string;           // 'naver-news', 'youtube', 'dcinside' 등
  collect(options: CollectionOptions): AsyncGenerator<T[], void, unknown>;
                         // ↑ 청크 단위로 yield → 메모리 효율 + 진행률 추적 가능
}

interface CollectionOptions {
  keyword: string;        // 검색 키워드
  startDate?: Date;       // 수집 시작일
  endDate?: Date;         // 수집 종료일
  maxItems?: number;      // 기사/영상 최대 수집량
  maxComments?: number;   // 댓글 최대 수집량
}

🔀 Map-Reduce 청킹 — 대량 데이터 처리
청킹 기준값
chunkTargetChars15,000자 (≈4K토큰)
minCharsForChunking20,000자 이상만 청킹
mapTimeoutMs300,000ms (5분)
reduceTimeoutMs300,000ms (5분)
분배 전략
기사: 라운드로빈 (시간순 보존)
댓글: 균등분배 (좋아요순 유지)
영상: 균등분배 
// 20,000자 이하 → 청킹 없이 단일 API 호출
if (totalChars < 20000) {
  return module.buildPrompt(input); // 단일 호출
}

// 20,000자 초과 → 청크 분할
const chunks = splitIntoChunks(input, 15000);

// Map 단계: 청크별 병렬 분석
const mapResults = await Promise.all(
  chunks.map(chunk => analyzeText(prompt, { timeoutMs: 300_000 }))
);

// Reduce 단계: 청크 결과 통합
const finalResult = await reduceResults(mapResults);

💡 별도 벡터 DB(Pinecone/Qdrant) 없이 기존 PostgreSQL에서 벡터 검색 — 운영 복잡도 최소화
pgvector SQL — 코사인 유사도 쿼리
SELECT a.id, a.title,
       1 - (a.embedding <=>
         '[0.12,-0.34,...]'::vector)
       AS similarity
FROM articles a
INNER JOIN article_jobs aj
  ON a.id = aj.article_id
WHERE aj.job_id = $1
  AND a.embedding IS NOT NULL
ORDER BY a.embedding <=>
  '[0.12,-0.34,...]'::vector
LIMIT 40;
<=> 연산자: 코사인 거리 (0=동일, 2=반대) · 1 - 거리 = 유사도

Worker 상태 판정 로직 (worker-health.ts)
🔴
down — workers 배열이 비어있음
🟢
healthy — active > 0 && idle < 60초
🟡
warn — active > 0 && idle ≥ 60초 (최근 활동 없음)
🟠
stuck — waiting/delayed > 0 && active = 0 (큐 적체)
⚪
idle — 모든 카운트가 0 (대기 중)
자동 복구 메커니즘 3종
① Stale Recovery (stale-recovery.ts)
Worker 시작 시 1회 실행 · 30분 이상 status='running' → status='failed'
에러: "Worker 재시작으로 인한 자동 복구 (30분 이상 running 상태)"
② Expire Paused (expire-paused.ts)
1시간 주기 cron · 24시간 초과 paused → cancelled
Stage Gate에서도 24h 초과 감지 → 자동 cancelled
③ BullMQ Stalled (워커 설정)
stalledInterval=300,000ms(5분) 주기 체크
maxStalledCount=2: 2회 자동 복구 → 3회째 failed
재시도 전략 (retry-utils.ts)
에러 유형최대 횟수대기 전략
Rate Limit (429) 5회 max(retryAfterSec×1000, attempt×3000)
Server Overload (503) 1회 고정 3000ms
Parse Error 2회 즉시 재시도
isRateLimitError(): /rate.*limit|429|RESOURCE_EXHAUSTED|[TR]PM/i
isServerOverloadError(): 503, overloaded, temporarily unavailable
isParseError(): "could not parse", "No object generated"
Stage Gate 동작 (pipeline-checks.ts)
breakpoints에 현재 stage 포함?
→ pausedAt 설정, status='paused'
→ polling: 3s → 10s → 60s (점진적)
