Vector DB — pgvector (PostgreSQL 16 Extension)
별도 Vector DB 없이 PostgreSQL에 pgvector 설치. multilingual-e5-small 384차원 임베딩으로 의미 기반 검색 구현
📐 임베딩 사양
모델: Xenova/multilingual-e5-small
@xenova/transformers (로컬 실행, GPU 불필요)
384차원 · 다국어 지원 (한국어 포함) · 평균 풀링 + 정규화
벡터 타입: vector(384)
Drizzle ORM 커스텀 타입 vector384
JS number[] ↔ pgvector [0.1,0.2,...] 자동 변환
거리 연산: 코사인 거리 (<=>)
유사도 = 1 - cosine_distance
ORDER BY embedding <=> queryVector::vector
저장 위치
articles.embedding vector(384)
comments.embedding vector(384)

🔍 Vector DB 활용 3가지 패턴
1. RAG Retrieval (분석 전처리)
파일: rag-retriever.ts
목적: 프롬프트 최적화 (토큰 절약)
방식: 키워드 임베딩 → pgvector 코사인 유사도 정렬 → topK 선별
모드:
• rag-light: 댓글 50개 필터
• rag-standard: 기사 30+10 · 댓글 30개
• rag-aggressive: 기사 15+5 · 댓글 15개
효과: 토큰 최대 60% 절감
2. 시맨틱 검색 (검색 기능)
파일: semantic-search.ts
목적: 자연어 질의로 관련 문서 검색
방식: 질의 텍스트 임베딩 → pgvector 정렬
필터: sentiment · jobId · minSimilarity
API: semanticSearch(), findSimilarDocuments()
기본값: topK=20, minSimilarity=0.4
3. 중복 제거 / 클러스터링 (전처리)
파일: deduplicator.ts, clusterer.ts
목적: 분석 입력 데이터 정제
방식: 기사 텍스트 즉석 임베딩 생성 → 코사인 유사도 비교
(DB 임베딩 미사용 — 실시간 계산)
deduplicator: 유사도≥임계값 시 긴 쪽 선택
clusterer: 대표 기사 1개 선별
📊 RAG SQL 예시
-- 키워드 관련성 기사 상위 30개 + 다양성 확보 10개
SELECT a.id, a.title, a.content, a.publisher, a.published_at, a.source,
       1 - (a.embedding <=> '[0.12,0.34,...]'::vector) AS similarity
FROM articles a
INNER JOIN article_jobs aj ON a.id = aj.article_id
WHERE aj.job_id = $jobId
  AND a.embedding IS NOT NULL
ORDER BY a.embedding <=> '[0.12,0.34,...]'::vector
LIMIT 40  -- articleTopK(30) + clusterRepresentatives(10)
-- minSimilarity 0.35 이상만 필터
Graph DB — 온톨로지 (PostgreSQL 기반 그래프)
Apache AGE 없이 PostgreSQL의 entities + relations 테이블로 그래프 구현. AI 분석 결과에서 엔티티/관계 추출 후 저장.
🕸️ 그래프 구조
노드 (entities) — 6가지 타입
• person — 인물 (정치인·연예인·임원)
• organization — 조직·기관·기업
• issue — 이슈·사건
• keyword — 핵심 키워드
• frame — 미디어 프레임
• claim — 주장·주제
엣지 (relations) — 6가지 타입
• supports — 지지/동의
• opposes — 반대/비판
• related — 연관
• causes — 원인-결과
• cooccurs — 공동 언급
• threatens — 위협
weight(0~1) · evidence(excerpt·moduleSource)
🔄 온톨로지 생성 파이프라인
①
AI 분석 완료 (Stage 1~4)
sentiment-framing · frame-war · risk-map · strategy 등의 결과에서 엔티티 추출
②
ontology-extractor.ts
JSONB 분석 결과 파싱 → ExtractedEntity[] + ExtractedRelation[] 추출
③
persist-ontology.ts
entities UPSERT (normalizedName+type 기준) → mentionCount 증가
relations INSERT (source+target+type 중복 방지)
④

GraphNode[] + GraphEdge[] 생성 → 프론트엔드 그래프 렌더링
📊 graph-builder.ts — 3가지 그래프 생성 함수
buildKeywordNetwork()
소스: sentiment-framing 결과
노드: topKeywords → 크기=count, 색=sentiment
엣지: relatedKeywords → 굵기=coOccurrenceScore
색상: positive=녹색, negative=빨강, neutral=회색
buildFrameWarGraph()
소스: frame-war + sentiment-framing
노드: dominant(파랑)·threatening(빨강)·reversible(노랑)·positive/negative(녹/주황)
엣지: threatens·conflicts·conflict 관계
목적: 프레임 경쟁 구조 시각화
buildRiskChainGraph()
소스: risk-map 결과
노드: topRisks → 크기=spreadProbability×40
색상: critical=암빨강, high=주황, medium=노랑, low=녹색
엣지: triggerConditions 공통 키워드로 연결
